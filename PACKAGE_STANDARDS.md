> **NORMATIVE DECISIONS (2026-06-11)** — This table is the single source of truth.
> Where any text below (or `dt_docs/DT-Flexibility.md`) conflicts with it, this
> table wins. Decided per `project-evaluation/02-gaps-and-required-changes/03-c-standards-consolidation.md`.
>
> | Question | Decision |
> |---|---|
> | Package manager / runtime | **pnpm + Node** (Bun is no longer part of the toolchain) |
> | Node engines | **>= 18** |
> | Output directory | **`dist/` flat** — `index.js` (ESM), `index.cjs` (CJS), `index.d.ts` / `index.d.cts` (types), built by tsup with `dts: true` |
> | Exports map | `types` / `import` / `require` / `default` pointing at the flat `dist/` paths above, plus `"./package.json"` |

> | Zod | `peerDependency >= 4.4.0`; zero Zod references in any `dist/**/*.d.ts` — enforced by `pnpm -r check-zod` (`scripts/check-zod.mjs`) |
> | Versioning | independent semver; release tooling to be selected in Phase C (changesets recommended) |

---

This document describes the conventions, structure, and rules used across all `km-*` npm packages.
Every package in this family follows these standards so that structure, tooling, and workflow
are predictable regardless of which package you are working on.

---

## 1. Runtime & Package Manager

| Tool | Version | Notes |
|------|---------|-------|
| **Bun** | latest | Primary runtime and package manager |
| **Node.js** | >= 14 | Minimum supported engine (declared in `package.json`) |
| **npm** | any | Used only for `npm publish` |

All day-to-day commands (`install`, `run`, `test`) use **Bun**. npm is only called for publishing.

## 2. Directory Layout

```
YOUR-PACKAGE-NAME/
├── src/                    # All source TypeScript
│   ├── index.ts            # Single flat public surface (re-exports everything)
│   └── <module>/           # One folder per domain (schemas/, adapters/, config/, …)
│       ├── index.ts        # Re-exports the module's public API
│       └── *.ts            # Implementation files
├── tests/                  # Vitest test suite — mirrors src/ structure
│   └── <module>/
│       └── *.test.ts
├── build/                  # Generated by tsup + tsc — never commit
│   ├── cjs/                # CommonJS output (.cjs)
│   ├── esm/                # ESM output (.mjs)
│   ├── js/                 # Universal JS output (.js)
│   └── types/              # Declaration files from tsc
├── .gitignore
├── .prettierrc
├── .release-it.json
├── vitest.config.ts
├── tsconfig.json
├── tsconfig.test.json
├── tsup.config.ts
├── package.json
├── LICENSE                 # MIT
├── README.md
├── CHANGES.md              # Human-written changelog
└── PACKAGE_STANDARDS.md   # This document — repository rules and conventions
```

### Rules

- `src/index.ts` is the **only** public entry point. No deep imports outside the package.
- Each sub-module has its own `index.ts` that re-exports everything — consumers always import from the top.
- `build/` is in `.gitignore`. It is never committed.
- `tests/` lives at the repo root, not inside `src/`.

## 3. package.json

### Required fields

```jsonc
{
  "name": "km-<name>",           // always prefixed km-
  "version": "x.y.z",           // semver, managed by release-it
  "description": "...",          // one clear, searchable sentence
  "license": "MIT",
  "author": "komeilm76",
  "homepage": "https://github.com/komeilm76/<name>#readme",
  "bugs": { "url": "https://github.com/komeilm76/<name>/issues" },
  "repository": { "type": "git", "url": "git+https://github.com/komeilm76/<name>.git" }
}
```

### Exports map (mandatory)

```jsonc
{
  // Legacy fields — read by old bundlers that don't understand "exports"
  "main":    "build/js/index.js",    // require() fallback
  "module":  "build/esm/index.mjs", // tree-shaking hint for older bundlers
  "types":   "build/types/index.d.ts",
  "typings": "build/types/index.d.ts",

  "exports": {
    ".": {
      // Bun reads raw TypeScript directly — no build step needed locally
      "bun":     "./src/index.ts",
      // TypeScript language server reads this for consumers' IDE support
      "types":   "./build/types/index.d.ts",
      // Node require() and CommonJS bundlers
      "require": "./build/cjs/index.cjs",
      // ESM import — Vite, Rollup, webpack 5, Node with "type":"module"
      "import":  "./build/esm/index.mjs",
      // Catch-all for environments that match none of the above
      "default": "./build/js/index.js"
    },
    // Sub-path exports let consumers explicitly request a specific format:
    // import ... from 'your-package/esm'
    "./esm": { "types": "./build/esm/index.d.mts", "import": "./build/esm/index.mjs" },
    "./js":  { "types": "./build/js/index.d.mts",  "default": "./build/js/index.js" },
    "./cjs": { "types": "./build/cjs/index.d.ts",  "require": "./build/cjs/index.cjs" }
  }
}
```

### files whitelist

```json
"files": ["build", "src", "README.md", "LICENSE"]
```

Only publish what consumers need. Never publish `tests/`, `coverage/`, or config files.

### Required scripts

| Script | Command |
|--------|---------|
| `fix:prettier` | Format all `src/**/*.{ts,json}` files |
| `fix` | Alias for `fix:prettier` |
| `dev` | `bun --watch run ./src/index.ts` |
| `build:types` | `tsc` — emit `.d.ts` only |
| `build:tsup` | `bun run build:types && tsup` |
| `build` | Full build (types + tsup) |
| `test` | `vitest run` |
| `test:coverage` | `vitest run --coverage` |
| `test:watch` | `vitest` |
| `publish-directly` | `npm publish --access public` |
| `auto-release` | `fix:prettier` + `build` + `test` + `release-it` |

`auto-release` is the **only** way to publish. It enforces formatting, a clean build, and passing
tests before the version bump and push.

## 4. TypeScript Configuration

### tsconfig.json (source)

Key settings:

| Option | Value | Reason |
|--------|-------|--------|
| `target` | `ESNext` | Bun and modern bundlers downlevel further as needed — ship the latest syntax |
| `module` | `Preserve` | Do NOT transform imports — tsup handles ESM/CJS conversion. Without this, tsc would silently rewrite `import` to `require()` |
| `moduleDetection` | `force` | Every file is treated as a module (own scope). Without this, files with no imports are treated as global scripts and their variables leak across files |
| `moduleResolution` | `bundler` | Matches how tsup and Bun resolve modules — allows `package.json` `exports` fields, no `.js` extension required on relative imports |
| `verbatimModuleSyntax` | `true` | Forces `import type` for type-only imports — prevents accidentally pulling in a value module just for its types |
| `allowImportingTsExtensions` | `false` | Prevents `import './foo.ts'` in source — published packages cannot rely on `.ts` extension stripping |
| `strict` | `true` | Full strict mode |
| `skipLibCheck` | `true` | Skips type-checking inside `node_modules` — broken third-party types won't fail your build |
| `noUncheckedIndexedAccess` | `true` | `arr[0]` returns `T \| undefined` — forces null checks for index access, catching out-of-bounds bugs at compile time |
| `noImplicitOverride` | `true` | Requires `override` keyword on subclass method overrides — prevents silent breakage when base class methods are renamed |
| `emitDeclarationOnly` | `true` | tsc only writes `.d.ts`; tsup handles all JS output |
| `outDir` / `declarationDir` | `build/types` | Type declarations land here |

### tsconfig.test.json (extends tsconfig.json)

Used only by the TypeScript language server in VSCode to provide IDE diagnostics for test files.
Vitest handles TypeScript natively at runtime and does not read this file.

```jsonc
{
  "compilerOptions": {
    "rootDir": ".",     // widen rootDir so tests/ is included
    "noEmit": true,
    "emitDeclarationOnly": false
  },
  "include": ["src/**/*.ts", "tests/**/*.ts"]
}
```

## 5. Build — tsup

Three output formats from a single config:

| Format | Output dir | Extension | Target | Consumed by |
|--------|-----------|-----------|--------|-------------|
| ESM | `build/esm/` | `.mjs` | `esnext` | Vite, Rollup, webpack 5, Node with `"type":"module"` |
| CommonJS | `build/cjs/` | `.cjs` | `es2019` | `require()`, older bundlers, Jest without ESM transform |
| Universal JS | `build/js/` | `.js` | `es2020` | `<script>` tags, CDNs (jsDelivr, unpkg), legacy environments |

**Why three formats?**
- `.mjs` — the `.mjs` extension explicitly signals ESM so bundlers never misread it as CJS.
- `.cjs` — the `.cjs` extension prevents ambiguity in projects where `"type":"module"` is set globally.
- `.js` — plain extension for maximum compatibility in environments that reject `.mjs`.

**Why different targets?**
- `esnext` for ESM: modern tools that consume `.mjs` will downlevel further themselves — ship the most compact output.
- `es2019` for CJS: covers Node 12+. Optional chaining (`?.`) and nullish coalescing (`??`) are ES2020 — esbuild downlevels them when this target is set.
- `es2020` for browser JS: covers all major browsers released after 2020 with no significant downlevelling needed.

**`clean` behaviour:** the first entry sets `clean: true` — it wipes `build/` at the start of the run. The other two set `clean: false` so they do not delete each other's already-written output.

Every format:
- Includes sourcemaps
- Emits `.d.ts` / `.d.mts` type declarations (via `dts: true`)
- Is minified

## 6. Tests — Vitest

### Configuration rules

- All test files live under `tests/` (not inside `src/`).
- File naming: `<module>.test.ts` — one file per source module is a reasonable target.
- Always import test utilities explicitly: `import { describe, it, expect } from 'vitest'`
- Set `environment` in `vitest.config.ts` to match the package target:

| Value | Use when |
|-------|----------|
| `'node'` | server-side or universal packages (default) |
| `'jsdom'` | client-side packages that need the DOM |
| `'happy-dom'` | lighter browser alternative to jsdom |

You can also override the environment per-file with a docblock: `// @vitest-environment jsdom`

- Coverage thresholds (minimum):

| Metric | Minimum |
|--------|---------|
| Branches | 80% |
| Functions | 85% |
| Lines | 85% |
| Statements | 85% |

Coverage reports go to `coverage/` (gitignored).

**Coverage provider `v8`** — uses Node's built-in V8 engine, no code instrumentation, fast and zero extra config. It is slightly less precise on untested conditional branches than `istanbul`. Switch to `istanbul` (add `@vitest/coverage-istanbul`) if you need exact branch coverage for complex conditional logic.

### Test style

- Use `describe` + `it` blocks.
- Each `it` tests one behaviour.
- No mocking of the module under test itself — test the real implementation.
- Keep test data minimal and inline.

## 8. Versioning — release-it

Releases are managed by **release-it** with the config in `.release-it.json`.

What `auto-release` does in order:

1. `fix:prettier` — formats the code
2. `build` — compiles types + JS bundles
3. `test` — must pass with zero failures
4. `release-it` — bumps version, commits, tags, pushes, creates GitHub release, publishes to npm

Commit message format for releases:

```
chore: release ( v1.2.3 )
```

Never manually edit `version` in `package.json`. Let release-it do it.

## 9. Git Conventions

### Commit messages

Follow Conventional Commits:

| Prefix | When |
|--------|------|
| `feat:` | New feature or exported API |
| `fix:` | Bug fix |
| `chore:` | Tooling, deps, config, CI |
| `refactor:` | Code change without behaviour change |
| `test:` | Adding or fixing tests |
| `docs:` | README, CHANGES, JSDoc only |

### Branches

| Branch | Purpose |
|--------|---------|
| `main` | Stable, always publishable |
| `<feature>` | Feature work — merge into main via PR or direct merge |

Never commit build artifacts to any branch.

## 10. Publishing Checklist

Before running `bun run auto-release`:

- [ ] All tests pass (`bun run test`)
- [ ] `CHANGES.md` updated with what changed in this version
- [ ] `README.md` reflects current API
- [ ] No uncommitted changes
- [ ] Logged in to npm (`npm whoami`)
- [ ] GitHub token available for release-it GitHub integration

## 11. Replacing Template Placeholders

When creating a new package from this template, replace every occurrence of:

| Placeholder | Replace with |
|-------------|-------------|
| `YOUR-PACKAGE-NAME` | actual package name, e.g. `km-validator` |
| `YOUR-GITHUB-USERNAME` | your GitHub username |
| `YOUR-KEYWORD-1` | relevant npm keywords |

Files to edit:
- `package.json` — name, description, homepage, bugs, repository, keywords
- `src/index.ts` — package doc comment and exports
- `README.md` — title, description, usage examples
- `CHANGES.md` — first version entry
- `LICENSE` — copyright name
- `vitest.config.ts` — set `environment` to `'node'` / `'jsdom'` / `'happy-dom'` to match your package target
- `tsup.config.ts` — set `platform` on each build entry to `'neutral'` / `'node'` / `'browser'`
